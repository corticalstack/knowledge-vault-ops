# Vault Structure

This document describes the physical layout of the knowledge vault repository and explains how files are organised. Understanding the structure matters because `verify-wiki.py` is scope-sensitive: it only validates files that live in specific subdirectories, and the distinction between wiki pages and infrastructure files determines what gets checked and what gets ignored.

## Top-Level Layout

```
knowledge-vault/
  AI/
  Cloud/
  Data/
  Engineering/
  Homelab/
  _INDEX.md          — cross-domain map and bridge concepts
  CLAUDE.md          — schema + instructions for Claude
  VAULT_GUIDE.md     — human-readable design guide
  scripts/
    verify-wiki.py
  docs/
    scripts/
      add_frontmatter.py
  graphify-out/
    GRAPH_REPORT.md
    graph.html
    graph.json
  luminary-scan/
    luminaries.json
    seen-urls.json
  projects/
    {repo-name}/
      context.md
      patterns.md
      _inbox/
  agents-shared/
    INDEX.md
    patterns.md
    mistakes.md
    _inbox/
  Templates/
    wiki-page.md
```

## Domain Structure

Each of the five domains (AI, Cloud, Data, Engineering, Homelab) follows the same internal pattern:

```
Domain/
  {category}/          — wiki pages live here
  _INDEX.md            — domain entry point
  _inbox/              — ingestion queue
  wiki-recent.base     — Obsidian view: all pages by updated date
  wiki-stubs.base      — Obsidian view: source_count ≤ 1
  wiki-mature.base     — Obsidian view: source_count ≥ 3
```

The `_INDEX.md` at the domain level is a curated entry point — a human-readable overview of what that domain covers and links to key pages. It is distinct from the root-level `_INDEX.md`, which maps connections across domains and tracks bridge concepts that span multiple areas.

The `_inbox/` directory is the drop point for the `vault-ingest` GitHub Action. Markdown files placed here are automatically processed, frontmatter is generated or updated, and the file is moved into the appropriate `{category}/` subdirectory. Nothing in `_inbox/` is treated as a finished wiki page.

The `.base` files are Obsidian Bases views — they define filtered, sorted views over wiki pages (by recency, maturity, stub status) and are read-only from a content standpoint. They do not contain wiki content.

### Valid Category Subdirectories per Domain

| Domain | Valid categories |
|--------|-----------------|
| AI | `concepts/` · `models/` · `techniques/` · `tools/` |
| Cloud | `services/` · `patterns/` · `architecture/` |
| Data | `concepts/` · `algorithms/` · `workflows/` · `tools/` |
| Engineering | `practices/` · `tools/` · `patterns/` |
| Homelab | `hardware/` · `os/` · `services/` |

These category names are not arbitrary — `verify-wiki.py` uses them to validate that each wiki page's `type` field is appropriate for the domain it lives in. A page in `AI/concepts/` must have `type: concept`; a page in `Cloud/services/` must have `type: service`.

## Wiki Pages vs Infrastructure Files

This distinction is the most important structural fact for tooling and automation.

**Wiki pages** are `.md` files that live inside a `domain/category/` subdirectory, for example `AI/concepts/Large Language Model.md` or `Engineering/practices/Trunk-Based Development.md`. These files must conform to the full wiki page schema (frontmatter + required body sections). `verify-wiki.py` validates only these files.

**Infrastructure files** are everything else. They are NOT validated by `verify-wiki.py`:

- `_INDEX.md` files at the root and domain level (cross-domain map and domain entry points)
- `_inbox/` directories and their contents at all levels
- `*.base` files (Obsidian views)
- `CLAUDE.md` and `VAULT_GUIDE.md` (operational documentation)
- `graphify-out/` (graph outputs generated by the graphify skill)
- `scripts/` and `docs/scripts/` (tooling)
- `luminary-scan/` (luminary URL tracking data)
- `projects/` and `agents-shared/` (repo-specific context and cross-repo patterns)
- `Templates/` (page templates)

If you run `verify-wiki.py` and a known-good infrastructure file produces an error, check that it has not accidentally been placed inside a `domain/category/` path. The validator does not distinguish by filename — only by location.

## Current Scale

At the time of the Day-0 bootstrap, the vault held approximately 4,000 notes organised into a flat structure. These were reorganised into the five-domain hierarchy described above. After the initial wiki compilation pass, the vault contained 83 wiki pages spread across all domains. Both figures grow continuously as `vault-ingest` and `luminary-scan` run — treat them as a baseline, not a stable count.
